  • When I wanted to find a healthy meal near me, I was tired of having to first check what's near me, then cross-reference menu items at each place comparing nutritional information. I wanted a tool where I can simply enter where I am, the maximum number of calories I want to consume that meal, and what cuisine I'm in the mood for and have it find specific items that meet my criteria.

  • I had to pull lots of food data from various sources and input that data into my database. I also needed the Google Maps API to cross-reference restaurant names that it found within my search radius with the restaurant names I entered into my database to pull up their nutritional nutritional information.

  • I found a crawler called Kimono that I used to crawl nutritional data from restaurant websites. I then used the Roo gem to pull the data from my spreadsheet and seed the database. I was able to enter approximately 30 restaurants worth of food data and then put the project on hold, realizing that gathering all of the nutritional data for the tens of thousands of restaurants out there was no easy task.

HoneyBee

  • Goal
  • Challenges
  • Solutions

  • This is a problem that I had faced as a small business owner. I needed an easy, affordable way to track applicants outside of email and spreadsheets. I didn't need an expensive HR Suite, just a place to create openings, receive applicants, and sort/filter through each one. I also wanted to store all past applicants in case a new position opened up, so that I can find one that matches what I'm looking for and reach out to him/her without posting a new opening. I needed all of the applicant information to be one page, and implement one-click accept/reject for efficient filtering.

  • When building, a challenge I faced was to avoid over-building. I had to constantly remind myself of what my goal was for this application - to create openings and track applicants. This was also my first time integrating a payment system, so that was a challenge for me as well.

  • I used the lessons from Getting Real, a book by the Basecamp team about the process of building web applications and staying lean. I also changed the way I go about building a feature: Instead of first coding Ruby, then the HTML, I reversed the order to first create sketches, then create the HTML, then the Ruby. This seemed counter-intuitive for me at first, but I realized it had allowed me to spend less time making changes and more time building what I truly needed. Regarding payments, I used Stripe, which offers some great documentation on how to implement it with a Rails app.
